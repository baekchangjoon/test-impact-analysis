# report.html 읽는 법 — TIA 인터랙티브 리포트 해설

`run-petclinic-tia.sh`가 마지막 단계에서 생성하는 자기완결형(self-contained) HTML 리포트
[`report.html`](report.html)를 화면 그대로 해설한다. 외부 의존성 없이 브라우저로 바로 열린다.
생성기는 [`make_html.py`](make_html.py)이고, 입력은 `testwise.json · scenarios.json · flaky.json ·
prod-files.txt`다.

> **한 줄 요약:** "어떤 변경이 어떤 테스트를 건드리는가"를 실제 per-test 커버리지로 측정해
> 보여준다. 모든 수치는 실측이며 목 데이터가 아니다.

---

## 헤더 — SUT 이름과 KPI

- **타이틀 / H1** — 대상 시스템(SUT)의 이름이 들어간다. 하드코딩이 아니라 파이프라인이 넘겨준
  `basename "$PETC"` 값이다(다른 프로젝트에 그대로 재사용 가능).
- **indexed commit** — per-test 커버리지를 DB에 저장한 베이스라인 커밋. 모든 `tia impact`
  판단의 기준점이다.
- **KPI 카드 4개**
  | 카드 | 의미 |
  |---|---|
  | black-box tests | 측정된 블랙박스 테스트 수 |
  | prod files covered | 스위트가 한 번이라도 실행한 프로덕션 파일 수 |
  | prod files blind | 어떤 테스트도 닿지 않은 프로덕션 파일 수(사각지대) |
  | (test·line) points | Σ(테스트별 커버 라인 수) — 커버리지 총량 |

## 공통 조작

- **헤더 클릭 = 정렬.** 모든 탭의 모든 컬럼이 오름차순/내림차순 토글된다(문자열은 가나다순,
  숫자는 크기순). 같은 헤더를 다시 누르면 방향이 뒤집히고, 화살표(▲/▼)는 **방금 클릭한
  컬럼에만** 표시된다. 막대 컬럼(`reach`, `fan-in`)도 그 막대가 나타내는 값으로 정렬된다.
- **행 클릭 = 펼치기.** 테이블 행을 누르면 상세(파일/테스트 목록)가 펼쳐진다. 칩 안의 링크를
  눌러도 행은 접히지 않는다.
- **검색창** — 테스트명/파일명으로 즉시 필터.

---

## 1 · Per-test impact range — "이 테스트는 무엇을 실행하는가"

각 블랙박스 테스트가 실제로 실행한 프로덕션 파일/라인.

| 컬럼 | 의미 |
|---|---|
| Test | 테스트 식별자 `클래스#메서드` |
| Result | 실행 결과(PASSED 등) |
| Files | 이 테스트가 건드린 프로덕션 파일 수 |
| Lines | 커버한 총 라인 수 |
| reach | Lines를 막대로 시각화(도달 범위) |

행을 펼치면 파일 칩(`파일명 N`, N=커버 라인 수)이 나온다. **파일명을 클릭하면 JaCoCo HTML
커버리지 리포트의 해당 파일 첫 커버 라인(`#L<n>`)으로 점프**한다(딥링크 대상은
`jacoco/` 디렉터리, 파이프라인이 함께 생성).

## 2 · Reverse selection index — "이 파일을 고치면 무엇을 돌려야 하는가"

탭 1의 역방향. 프로덕션 파일 → 그것을 실행하는 테스트 목록.

| 컬럼 | 의미 |
|---|---|
| Production file | 프로덕션 파일 경로 |
| #tests | 이 파일을 실행하는 테스트 수(fan-in) |
| fan-in | #tests를 막대로 시각화 |

**fan-in이 클수록 변경 위험이 큰 핫스팟**이다(고치면 많은 테스트가 영향). 행을 펼치면 테스트
칩이 나오고, **테스트명을 클릭하면 로컬 테스트 소스 파일(`file://…Test.java`)이 열린다.**

## 3 · tia impact — "이 diff는 어떤 테스트를 선별하는가"

`tia impact`의 핵심 산출물. **무엇과 무엇을 비교하는지**가 중요하다:

- **베이스라인** = 헤더의 indexed commit(= `tia index`로 커버리지를 저장한 시점).
- **비교 대상** = 그 베이스라인에 **적용하려는 변경(diff)**. `origin ↔ local` 같은 고정된 비교가
  아니라, 평가하려는 변경이면 무엇이든 된다 — 커밋 전 워킹트리, 브랜치 vs 베이스라인, PR diff 등.
- **이 데모에서는** 시나리오마다 대상 파일/라인에 `// tia-probe` 한 줄을 넣은 **합성 diff**를
  베이스라인과 비교한 것(perturb-and-revert — 워킹트리 변경은 즉시 복원).
- **선별 규칙** = diff의 변경 라인 ∩ 각 테스트의 커버 라인 ≠ ∅ → 선별.

용어:

| 배지 / 표기 | 영문 | 뜻 |
|---|---|---|
| 🎯 정밀 선별 | DETERMINISTIC | 변경 라인이 그 테스트의 커버리지에 **정확히 포함**됨 → 직접 영향받는 테스트 |
| 🛡 보수적 선별 | CONSERVATIVE | 변경을 특정 커버리지에 **매핑할 수 없어**(새 파일·설정 변경 등) 누락을 막으려 **안전하게 후보로 포함**(fallback) |
| ▼ NN% 절감 | reduction | 전체 테스트 중 **실행을 건너뛸 수 있는 비율**. 높을수록 TIA 이득이 큼 |
| 🛡 safe fallback | | 보수적 선별로 전체를 포함 — 절감은 없지만 누락 위험 0 |
| ⚠ 0개 선별 | blind spot | 어떤 테스트도 닿지 않는 파일을 변경 → 선별 0개(탭 5 참조) |

각 카드는 시나리오 id·제목·절감률·diff·선별된 테스트를 보여준다. 보수적/사각지대 카드는
테두리 색(노랑/빨강)으로 구분된다.

## 4 · Flaky detection — "스위트가 흔들리는가 / 검출기는 살아있는가"

**두 카드는 서로 다른 것을 측정한다. 값이 같을 필요가 없다.**

- **① 실측 (Real)** — 동일 스위트를 N회(기본 5회) **실제로** 돌려, 매번 결과(Pass/Fail)가 바뀌는
  테스트가 있는지 측정. `0.0% (0/24)`이면 24개 모두 N번 내내 같은 결과 → **결정론적**.
- **② 검출기 검증 (synthetic · 양성 대조군)** — 실제 실행이 **아니다.** 한 테스트의 결과를
  **일부러 Pass→Fail→Pass로 뒤집은** 가짜 실행 3개를 만들어 flaky 검출기에 넣는다. 검출기가
  이 1개를 flaky로 **집어내면 정상**(조작했으니 당연히 잡혀야 한다). `4.2% (1/24)`.

→ ②가 조작된 1개를 정확히 잡아낸다는 사실이, **①의 0%가 "검출기 고장으로 0"이 아니라
"진짜 안정적이라 0"임을 보증**한다(positive control).

ratio는 **퍼센트와 분수를 함께** 표기하며(`4.2% (1/24)`), flaky로 표시된 테스트명은 로컬
소스 파일 링크다.

## 5 · Coverage blind spots — "아무도 안 지키는 파일"

블랙박스 스위트가 한 번도 실행하지 않은 프로덕션 파일 목록. 이 파일들을 **수정**하면 정밀
선별(DETERMINISTIC)에서 0개가 나온다(탭 3의 시나리오 E). 블랙박스 스위트는 `/api/**` REST
계층을 지키지만 MVC/Thymeleaf·일부 보안 인프라는 닿지 않음을 드러낸다.

## 로컬 링크에 관한 주의

탭 1의 JaCoCo 딥링크와 탭 2·4의 소스 열기 링크는 **생성한 머신의 절대경로** 기준이다. 커밋된
`report.html`은 샘플이며, 본인 환경에서 `run-petclinic-tia.sh`를 실행하면 올바른 경로로
재생성된다. JaCoCo 딥링크는 같은 실행에서 만들어진 `petclinic-demo/jacoco/`(gitignore됨)가
옆에 있어야 동작한다.

## 재현 / 회귀

```bash
bash petclinic-demo/run-petclinic-tia.sh        # 전체 파이프라인 재생성
python3 -m unittest discover -s petclinic-demo -p 'test_*.py'   # 리포트 생성기 회귀 테스트
```

리포트 생성기의 사용자 계약(타이틀 SUT화, 전 컬럼 정렬, JaCoCo 딥링크, 테스트 소스 링크,
impact 용어, flaky 퍼센트+분수, `</script>` 인젝션 방어)은
[`test_make_html.py`](test_make_html.py)가 검증하며, CI(`.github/workflows/ci.yml`의
`report-generator` 잡)에서 매 PR 회귀로 돌린다.
